# 7.11 — Performance and Load Test Implementation

## Description

Prompt to generate executable performance, load, stress, and benchmark test code from the design defined in `07-06`. Produces scripts ready to run with the project's load testing tool.

**When to use:** after completing the performance test design (`07-06`), when the test environment is already running (QA / staging) and the executable script for the selected tool is needed.

---

## Required prior context

> Include the block from `00-framework.md` before this prompt.
> Attach the **Test stack profile** generated by `07-00` (run it once per project if it does not yet exist).
> Attach the performance test design generated by `07-06` with the scenarios, thresholds, and selected tool.

---

## Full prompt

```text
Objective:
Implement the executable performance and load test scripts defined in the attached
07-06 design, using the project's load testing tool.

Steps:

1. CONTEXT IDENTIFICATION
   From the 07-06 design and 07-00 profile, extract:
   - selected load testing tool (k6 / Locust / JMeter / Artillery / hey / wrk)
   - endpoints and operations to test with exact parameters (URL, method, headers, payload)
   - required test types: load / stress / spike / soak / benchmark
   - acceptance thresholds per scenario (P95, P99, error rate, throughput)
   - authentication: do endpoints require a token? how to obtain it?
   - test data: what synthetic data or fixtures are needed?

2. SCRIPT STRUCTURE
   Generate a well-structured script that includes:
   
   a) Global configuration:
      - base URL of the test environment (environment variable — never hardcoded)
      - connection and response timeouts
      - acceptance thresholds (thresholds) as code — the script must fail if any threshold is violated
      - labels to identify scenarios in the report
   
   b) Authentication (if applicable):
      - setup function to obtain a token before running the tests
      - token renewal handling if the test is long-running (soak)
   
   c) Synthetic test data:
      - generation of unique users / IDs / payloads per virtual user
      - avoid collisions between concurrent users
      - never use real production data
   
   d) Load scenarios:
      For each scenario from the 07-06 design, implement:
      - test function with HTTP request + response validation
      - expected status code verification
      - response time verification within threshold
      - error handling: log and continue, do not stop the entire test
      - realistic think time between requests (simulate human behavior)

3. TEST TYPES TO IMPLEMENT
   
   a) Load test (normal load):
      ```
      Ramp-up: 0 → N users in X minutes
      Sustained: N users for Y minutes
      Ramp-down: N → 0 in X minutes
      ```
      Implement with the selected tool's syntax.
   
   b) Stress test (overload):
      ```
      Progressively increase load until the breaking point is found
      Record at what load level thresholds start to fail
      ```
   
   c) Spike test (sudden spike):
      ```
      Normal load → sudden spike (10x) for 30 seconds → back to normal load
      Verify the system recovers without persistent degradation
      ```
   
   d) Soak test (endurance):
      ```
      Sustained load at 70% of maximum for a long period (1-8 hours)
      Detect: memory leaks, gradual response time degradation,
      DB connection exhaustion, session accumulation
      ```
   
   e) Benchmark (baseline):
      ```
      Simple measurement script for a critical endpoint — run before and after a change
      Compare P50, P95, P99 to detect performance regressions
      ```

4. THRESHOLDS AND FAILURE CRITERIA
   Encode thresholds directly in the script:
   - the script must return a non-zero exit code if any threshold is violated
   - minimum recommended thresholds if not specified in the design:
     * P95 response time < 500ms for APIs
     * P99 response time < 1s for APIs
     * error rate < 0.1%
     * throughput ≥ expected in production

5. CI/CD INTEGRATION
   Provide the exact command to run each test type:
   - environment variable for base URL: `BASE_URL=https://staging.example.com`
   - exit flags for pipeline integration
   - command to generate readable report (HTML, JSON, InfluxDB)
   - instruction on when to run each test type in CI/CD:
     * benchmark: on every PR with changes to critical endpoints
     * load test: before every release to production
     * stress/spike/soak: periodically (weekly / monthly)

6. SYNTHETIC DATA AND FIXTURES
   If the test requires input data:
   - generate helpers to create unique data per virtual user (avoid conflicts)
   - provide a test data setup script in the environment (if applicable)
   - provide a teardown script to clean data after the test

CONSTRAINTS:
- use exclusively synthetic data — never real production data
- scripts must run without manual intervention (headless)
- base URLs must be configurable via environment variables
- the script must be idempotent: runnable multiple times without unintended cumulative effects
- do not run stress/spike/soak against production without a maintenance window

Deliverables:
- executable scripts by test type (load, stress, spike, soak, benchmark),
- execution command for each type with environment variables,
- proposed CI/CD pipeline integration,
- guide for interpreting results: which metrics to review and how to read them.
```

## Expected output

### Load test script (k6 example)

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

const errorRate = new Rate('errors');
const responseTime = new Trend('response_time', true);

export const options = {
  stages: [
    { duration: '2m', target: 50 },   // ramp-up
    { duration: '5m', target: 50 },   // sustained
    { duration: '2m', target: 0 },    // ramp-down
  ],
  thresholds: {
    http_req_duration: ['p(95)<500', 'p(99)<1000'],
    errors: ['rate<0.01'],
  },
};

const BASE_URL = __ENV.BASE_URL || 'http://localhost:8000';

export function setup() {
  const res = http.post(`${BASE_URL}/auth/token`, JSON.stringify({
    username: 'test_user',
    password: __ENV.TEST_PASSWORD,
  }), { headers: { 'Content-Type': 'application/json' } });
  return { token: res.json('access_token') };
}

export default function (data) {
  const params = { headers: { Authorization: `Bearer ${data.token}` } };
  const res = http.get(`${BASE_URL}/api/products`, params);

  const success = check(res, {
    'status 200': (r) => r.status === 200,
    'response time < 500ms': (r) => r.timings.duration < 500,
  });

  errorRate.add(!success);
  responseTime.add(res.timings.duration);
  sleep(1);
}
```

### Execution commands

```bash
# Load test
BASE_URL=https://staging.example.com TEST_PASSWORD=xxx k6 run --out json=results/load.json tests/performance/load-test.js

# Benchmark (before and after the change)
BASE_URL=https://staging.example.com k6 run tests/performance/benchmark.js

# Stress test
BASE_URL=https://staging.example.com k6 run tests/performance/stress-test.js
```

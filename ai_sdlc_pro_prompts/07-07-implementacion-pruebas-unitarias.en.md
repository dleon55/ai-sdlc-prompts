# 7.7 — Unit Test Implementation

## Description

Prompt to generate executable unit test code from the design defined in `07-01`. Produces test files ready to run in the project's test framework.

**When to use:** after completing unit test design (`07-01`), when the agent has access to the source code and the project's test framework.

---

## Required prior context

> Include the block from `00-framework.md` before this prompt.
> Attach the **Test stack profile** generated by `07-00` (run it once per project if it does not exist yet).
> Attach or reference the test matrix generated by `07-01`.

---

## Full prompt

```text
Objective:
Implement the unit tests defined in the attached design matrix using the project's test framework.

Steps:
1. Identify the active test framework in the repository
   (package.json, pyproject.toml, Gemfile, pom.xml, etc.).
2. For each case in the matrix, generate the corresponding test code:
   - descriptive test name following project conventions,
   - arrange: input data setup and required mocks,
   - act: invocation of the unit under test,
   - assert: validation of the expected result.
3. Group tests by unit or module in the corresponding test file.
4. Follow the project's naming conventions and directory structure.
5. Include mocks or stubs where the unit depends on external services,
   database, or I/O.

Constraints:
- Do not modify source code, only test files.
- Keep each test independent with no side effects between them.
- Prioritize cases marked as critical or required coverage.

Deliverables:
- complete and executable test files,
- verified execution command for the project environment,
- estimated coverage achieved by the generated tests.
```

---

## Standard formula usage

```text
Use the unit test implementation prompt and adapt it to:
- repository: [NAME OR URL]
- issue or requirement: [REFERENCE]
- branch: [WORKING BRANCH]
- test framework: [AUTO-DETECT / pytest / Jest / Vitest / JUnit / RSpec / etc.]
- test matrix: [ATTACH OR REFERENCE THE 07-01 DOCUMENT]
- documents to review: source code of units under test, framework configuration
- specific output goal: executable test files with ≥ 80% coverage
- depth level: high
```

---

## Expected output

Test files in the project's test structure, for example:

```
tests/unit/test_[module].[ext]
src/[module]/[module].test.[ext]
spec/unit/[module]_spec.[ext]
```

With internal AAA structure (Arrange / Act / Assert):

```
[test class or suite for the unit]
  ✓ [main positive scenario]
  ✓ [expected negative scenario]
  ✓ [identified edge case]
```

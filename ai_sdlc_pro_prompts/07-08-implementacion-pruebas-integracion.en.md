# 7.8 — Integration Test Implementation

## Description

Prompt to generate executable integration test code from the design defined in `07-02`. Validates real interaction between modules, services, APIs, and the database using the project's test framework.

**When to use:** after completing integration test design (`07-02`), when the test environment (database, dependent services) is available or can be simulated with containers or stubs.

---

## Required prior context

> Include the block from `00-framework.md` before this prompt.
> Attach the **Test stack profile** generated by `07-00` (run it once per project if it does not exist yet).
> Attach or reference the integration test plan generated by `07-02`.

---

## Full prompt

```text
Objective:
Implement the integration tests defined in the attached design plan using the project's
test framework.

Steps:
1. Identify the test framework and integration tools in the repository
   (pytest + fixtures, Jest + supertest, Testcontainers, WireMock, etc.).
2. For each flow in the plan, generate the corresponding test code:
   - initial state setup: database, external services, or required stubs,
   - execution of the complete integration flow,
   - validation of the result across all involved components,
   - teardown or cleanup of the generated state.
3. Implement reusable fixtures or helpers for common test data.
4. Use stubs or test containers for external dependencies not available in CI.
5. Follow the project's structure conventions for integration tests.

Constraints:
- Isolate each test's state to prevent interference between runs.
- Do not call real production services; use test environments or doubles.
- Include explicit error handling and timeouts for service calls.

Deliverables:
- complete and executable integration test files,
- test environment setup instructions (environment variables, required services),
- verified execution command,
- notes on external dependencies requiring additional configuration.
```

---

## Expected output

Test files in the project's test structure, for example:

```
tests/integration/test_[flow].[ext]
src/[module]/__tests__/[flow].integration.test.[ext]
spec/integration/[flow]_spec.[ext]
```

With internal structure:

```
[integration suite for the flow]
  setup: [environment initialization]
  ✓ [complete flow positive scenario]
  ✓ [flow with error in component X]
  ✓ [flow with boundary data]
  teardown: [state cleanup]
```
